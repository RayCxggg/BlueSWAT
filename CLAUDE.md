fixed broken policies, added the LL TX hook, regenerated bytecodes, built

## GATT patch service UUIDs (`peripheral/src/firewall_patch.c`)

128-bit, base `bd00****-7e57-49a1-a0bd-e6cf8d00f1ed`:

| Characteristic | UUID first 32 bits | Permission | Purpose |
|---|---|---|---|
| Service | `bd000001` | — | Primary service container |
| Header | `bd000002` | write, 2 bytes `[class, type]` | Set target FSM slot, reset staging |
| Body | `bd000003` | write, N bytes per write, appends | Accumulate bytecode chunks |
| Commit | `bd000004` | write, 1 byte | Call `ifw_install_policy` with staged bytes |
| Status | `bd000005` | read, returns last `s32_t` rc | Read install result code |

## Residual gaps (paper vs. artifact, after current work)

These are documented for honesty; not all are bugs.

- **No eBPF verifier**: any bytecode passed to `ifw_install_policy()` runs
  in the VM. A malicious or buggy program could loop forever, OOB read,
  etc. The interpreter has `bounds_check_enabled` but it's off by default.
- **`conn_chan_hop` and `scan_rsp_len` are stateless**: they're per-packet
  range checks. Genuine state-awareness only exists for the 4 stateful
  policies (dc_nesn, llcp_len_req, llcp_conn_param_req, smp_ident_check)
  plus the multi-point `conn_chan_map` / `lll_interval`.
- **Central role uncovered**: `ifw_central_setup` is empty. Per-paper this
  is "peripheral filter" only.
- **SPI HCI policies (CVE-2020-10065)** unregistered on this board (no
  split-SoC SPI HCI driver active). Bytecodes still in the tree.
- **One static `result` shared between LL RX and LL TX paths**: pre-existing
  artifact concern; could race if mayfly is preempted. Not introduced by
  the current work.
